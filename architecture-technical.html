技术架构说明

律所数字化系统 — 技术架构

模块化单体 + 独立 AI/飞书网关 + 清晰数据分层。前端 React 18，后端 NestJS + TypeORM，AI 通过独立 Hermes Agent 服务接入，飞书通过 Python WebSocket Consumer 接入。

关键数据流

基于当前代码实现的主要交互路径，非目标规划。

📝 文本消息流（飞书）

  1. 用户在飞书发文本
  2. FeishuConsumer 通过 WebSocket 接收
  3. 调用 Hermes Agent /v1/responses
  4. Agent 调用 law-cli → 后端 /api/v1/cli/*
  5. 读操作直接返回文本；写操作发送确认卡片

📎 文件消息流（飞书）

  1. 用户上传文件到飞书
  2. FileUploadHandler 本地下载
  3. Hermes 识别文件类型与当事人
  4. POST /cli/feishu/file-intent
  5. Backend 返回上传成功/补信息/选案件

✅ 写操作确认流

  1. Agent 调用 case create/update/stage advance
  2. CLI 返回 pending: true
  3. Backend 发送确认卡片到飞书
  4. 用户点击确认 → /webhook/feishu/card-action
  5. CardActionHandler 执行实际写操作

⏰ 提醒推送流

  1. ReminderTriggerService Cron 扫描
  2. 查询 case_reminders pending 状态
  3. ReminderDispatchService 按渠道分组
  4. 飞书通道调用 FeishuService.sendMessage()
  5. 用户点击卡片按钮标记已处理/忽略

鉴权体系

不同入口采用不同的认证方式，CLI 通道按平台区分权限。

Web UI

JWT Bearer → AuthGuard → PermissionGuard(@RequirePermissions) → Controller

Agent Chat

JWT Bearer → Gateway 鉴权 → Socket.IO /agent-hub

CLI / 飞书

x-cli-token: fs_xxx → CliAuthGuard + x-cli-user-id

CLI / 企微

x-cli-token: wx_xxx → 只读权限

CLI / Web JWT

x-cli-token: jwt_xxx → 按 RBAC 角色权限

BackendConsumer

Authorization: Bearer {BACKEND_WS_TOKEN} → Gateway Consumer 鉴权

架构决策说明

基于当前实现的真实决策，非早期规划文档。

为什么选择模块化单体？

  1. 年案件量 300-500 件规模，单机足以支撑
  2. 模块间通过 Service 注入解耦，保留拆分独立服务的可能
  3. 降低运维复杂度，减少分布式事务成本

为什么 AI 走独立服务？

  1. Hermes Agent 独立于后端，通过 Socket.IO / HTTP 与后端通信
  2. 飞书消息由 FeishuConsumer 直接调 Agent，Web 消息由 BackendConsumer 调 Agent
  3. 模型迭代与业务后端升级互不阻塞
